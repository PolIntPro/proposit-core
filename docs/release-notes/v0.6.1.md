# 0.6.1

## Lenient parser mode

`ArgumentParser.build()` now accepts an optional `{ strict?: boolean }` second
argument. When `strict` is `false`, recoverable errors are collected as warnings
instead of throwing, allowing partially valid LLM output to produce a usable
engine.

### Warning codes

Each skipped entity emits a structured `TParserWarning` with one of:

- `FORMULA_PARSE_ERROR` — malformed formula syntax (premise skipped)
- `FORMULA_STRUCTURE_ERROR` — structural violation such as nested implies
  (premise skipped)
- `UNDECLARED_VARIABLE_SYMBOL` — formula references a variable symbol with no
  matching declaration (premise skipped)
- `UNRESOLVED_CLAIM_MINIID` — variable references a nonexistent claim miniId
  (variable skipped; premises that depend on it are also skipped)
- `UNRESOLVED_SOURCE_MINIID` — claim references a nonexistent source miniId
  (association skipped)
- `UNRESOLVED_CONCLUSION_MINIID` — conclusion premise miniId doesn't match any
  surviving premise (conclusion falls back to auto-assignment)

### New types

- `TParserWarning` — `{ code, message, context }` warning descriptor
- `TParserWarningCode` — union of the six warning code literals
- `TParserBuildOptions` — `{ strict?: boolean }` options for `build()`

### CLI changes

- The `parse` command now uses lenient mode (`strict: false`) and prints any
  warnings to stderr.

## MiniId prefix conventions in LLM prompt

The system prompt now instructs LLMs to use distinct prefixes for each entity
type (`c1`, `s1`, `v1`, `p1`) to reduce cross-reference confusion in generated
output.

## Bug fix

- **Parser auto-normalization** — `ArgumentParser.build()` now creates the
  `ArgumentEngine` with `autoNormalize: true`, so formula buffer nodes are
  automatically inserted between nested operators (e.g., `(P and Q) or P`).
  Previously, valid formulas with nested `and`/`or` operators threw because the
  PEG parser doesn't emit explicit formula AST nodes for parenthesized
  sub-expressions.
